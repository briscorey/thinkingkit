## How it works

Horst Rittel and Melvin Webber coined "wicked problems" to describe a class of problems fundamentally different from "tame" ones. Tame problems (like solving an equation) have clear formulations, definitive solutions, and objective tests for correctness. Wicked problems have none of these.

Characteristics of wicked problems: there is no definitive formulation (the problem changes as you understand it better), there is no stopping rule (you can always do more), solutions are not true or false but better or worse, every attempt to solve the problem changes the problem, and there is no opportunity to learn by trial and error (every intervention counts).

Climate change, poverty, education reform, and urban planning are classic wicked problems. So are many personal and organisational challenges: team culture, work-life balance, and relationship dynamics.

## Real-world examples

**Education.** "How do we improve education?" is a wicked problem. Every proposed solution (more testing, less testing, more technology, smaller classes) changes the system in ways that create new problems. There's no objective measure of "solved."

**Organisational culture.** You can't fix culture the way you fix a bug. Every intervention changes the dynamics, and the "problem" is defined differently by every stakeholder.

## When to use it

Recognise wicked problems so you can stop applying tame-problem tools to them. If you're trying to "solve" something that keeps shapeshifting, you may be facing a wicked problem — and the appropriate response is iterative intervention and monitoring, not searching for the definitive solution.

## Common mistakes

The main mistake is treating a wicked problem like a tame one — searching for the "root cause" and the "solution." Wicked problems don't have root causes. They have interacting dynamics. The second mistake is paralysis — concluding that because the problem can't be "solved," nothing should be done.

## Try it now

Think of a persistent challenge in your work or life. Is it tame (clear definition, testable solution) or wicked (changes as you engage with it, no definitive solution)? If wicked, are you using tame-problem tools? What would change if you treated it as an ongoing practice rather than a puzzle to solve?
